# SEO-Анализатор для macOS

Нативное приложение для macOS на **SwiftUI**, которое выполняет комплексный SEO-аудит сайта по принципу [SEOptimer](https://www.seoptimer.com). Интерфейс полностью на русском языке, с возможностью выгрузки отчёта в **PDF** и **Markdown**.

## Возможности

- Ввод адреса сайта и анализ в один клик.
- Реальные проверки (не заглушки): приложение загружает страницу, HTTP-заголовки, `robots.txt` и `sitemap.xml`, измеряет время ответа и разбирает HTML.
- Пять категорий оценки, как в SEOptimer, с буквенными оценками **A+ … F**:
  - **SEO** — Title, Meta Description, H1, структура заголовков, ALT-атрибуты, объём контента, canonical, robots.txt, sitemap.xml, индексация, язык страницы.
  - **Производительность** — время ответа, размер HTML, сжатие (gzip/brotli), число JS/CSS, инлайн-стили, кэширование, DOCTYPE.
  - **Удобство использования** — мобильная адаптация (viewport), favicon, иконка для iOS, кодировка, читаемость контента.
  - **Соцсети** — Open Graph, Twitter Card, ссылки на профили (Facebook, X, Instagram, LinkedIn, YouTube, VK, Telegram).
  - **Безопасность** — HTTPS/SSL, HSTS, X-Content-Type-Options, защита от кликджекинга, Content-Security-Policy, раскрытие версии сервера.
- Итоговая оценка как взвешенное среднее по категориям.
- Подробные рекомендации по каждому пункту.
- Экспорт в **PDF** (многостраничный, через CoreText) и **Markdown**.

## Технологии

- Swift 5 / SwiftUI
- `URLSession` с `async/await` для сетевых запросов
- CoreText для генерации PDF
- Без сторонних зависимостей

## Сборка и запуск

1. Откройте `SEOAnalyzer.xcodeproj` в Xcode 16 или новее (macOS 13+).
2. При необходимости выберите свою команду подписи в настройках таргета (для локального запуска подходит автоматическая подпись).
3. Нажмите **Run** (⌘R).

## Как пользоваться

1. Введите адрес сайта (например, `example.com`) в поле вверху.
2. Нажмите **«Анализировать»**.
3. Изучите итоговую оценку и разбивку по категориям.
4. Нажмите **«Сохранить PDF»** или **«Сохранить Markdown»** для выгрузки отчёта.

## Автосборка и релизы

В репозитории настроен GitHub Actions ([`.github/workflows/release.yml`](.github/workflows/release.yml)), который собирает готовое приложение и публикует его в разделе **Releases**:

- **Автоматически** — при пуше тега вида `v1.2.3`:
  ```bash
  git tag v1.1.0
  git push origin v1.1.0
  ```
- **Вручную** — вкладка **Actions → Сборка и релиз macOS-приложения → Run workflow**, указав версию.

Workflow собирает `SEOAnalyzer.app` на раннере **macos-26** (Xcode 26/27), делает ad-hoc подпись и собирает **DMG-установщик** `SEOAnalyzer.dmg` (перетащить в «Программы»), который прикладывается к релизу. Версия (`CFBundleShortVersionString`) подставляется из тега.

## Установка

1. Скачайте `SEOAnalyzer.dmg` из последнего [релиза](https://github.com/Arti-Ko/SEOAnalyzer/releases), откройте его и перетащите приложение в **«Программы»**.
2. Если macOS пишет, что **«программа повреждена»** — это Gatekeeper: приложение не нотаризовано в Apple (нет платного Developer-аккаунта). Снимите «карантин» одной командой и запустите:
   ```bash
   xattr -cr /Applications/SEOAnalyzer.app
   ```
   Либо: правый клик по приложению → **Открыть** → **Открыть**.

> Полностью убрать это предупреждение можно только подписью *Developer ID* + нотаризацией Apple (платный аккаунт $99/год). Это ограничение Apple, а не сборки.

## Проверка и автообновление

Приложение умеет обновляться само:

- При запуске (если включено в **Настройках**, ⌘,) выполняется тихая проверка последнего релиза на GitHub.
- Проверить вручную можно кнопкой ⟳ в шапке или через меню **SEO-Анализатор → Проверить обновления…** (⌘U).
- Если доступна новая версия, приложение показывает заметки к релизу и кнопку **«Обновить»**: оно скачивает `SEOAnalyzer.dmg`, монтирует его, заменяет свой бандл, снимает «карантин» и автоматически перезапускается.

> Самообновление требует записи в собственный бандл, поэтому App Sandbox в проекте отключён (приложение распространяется через GitHub, а не Mac App Store).

## Заметки о точности

Все показатели вычисляются на основе фактически загруженной страницы и её заголовков. Часть проверок (например, реальная скорость рендеринга в браузере или полный аудит сторонних ресурсов) выходит за рамки серверного анализа HTML и оценивается косвенно — по размеру разметки, числу подключаемых ресурсов и заголовкам кэширования/сжатия.
